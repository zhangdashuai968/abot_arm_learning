## 学习目标

理解 ROS 中的\*\*服务通信（Service）\*\*机制。Topic 适合传感器数据流（异步、单向），但如果你需要“一问一答”的逻辑（比如：计算加法、触发拍照、重置环境），就需要 Service（同步、双向）。

***

## 关键问题及答案

### 1. Topic 和 Service 的根本区别是什么？如果我用 topic 实现一问一答，会碰到什么问题？

**答**：

- **根本区别**：Topic 是“发广播”，异步且单向；Service 是“打电话”，同步且双向。
- **用 Topic 的问题**：
  - **无法对应**：如果你连发两个请求，收到的回复不知道对应哪一个。
  - **无法确认**：你不知道对方是否真的收到了，除非再开一个反馈 Topic，导致系统极其臃肿。

### 2. Service 的 `.srv` 文件用 `---` 分隔，`.msg` 文件不用。为什么？

**答**：
`.srv` 定义的是一个**交互过程**。`---` 是分界线：

- **上方（Request）**：客户端发出的“考卷”（输入参数）。
- **下方（Response）**：服务端返回的“答题卡”（输出结果）。
  `.msg` 只是单纯的数据包，不需要区分方向。

### 3. Server 端回调函数返回 `false` 时，Client 端会抛异常吗？

**答**：
**不会抛出异常**。但 Client 端的 `client.call(srv)` 函数会返回 `false`。因此，写 Client 代码时必须检查返回值，不能盲目读取结果。

### 4. 如果 Client 调用时 Server 还没启动，会发生什么？

**答**：
会直接报错退出。**优雅处理**：在代码中使用 `ros::service::waitForService("name")`。这能让 Client 挂起等待，直到 Server 准备好，提高系统的鲁棒性。

### 5. 为什么不能用 Service 来传 30Hz 的激光雷达数据？

**答**：

- **开销太大**：Service 每次都要“握手-传输-断开”，高频调用会榨干 CPU。
- **容易卡死**：Service 是同步阻塞的。如果 Server 处理慢了，Client 就会卡住，导致实时感知系统崩溃。

## 核心知识点总结

### 1. 通信模型对比

| 维度     | Topic (话题) | Service (服务) |
| :----- | :--------- | :----------- |
| **模式** | 异步、单向、多对多  | 同步、双向、一对多    |
| **类比** | 广播电台 / 发传单 | 函数调用 / 打电话   |
| **特点** | 只要发了就不管了   | 必须等对方回信才继续   |
| **场景** | 传感器数据、控制指令 | 状态查询、动作触发、计算 |

### 2. 编程要素

- **服务端 (Server)**：
  - **核心**：编写**回调函数**（处理 Request，填好 Response）。
  - **注册**：`nh.advertiseService("name", callback)`。
  - **挂起**：必须有 `ros::spin()`，否则服务无法响应请求。
- **客户端 (Client)**：
  - **核心**：创建客户端对象 `nh.serviceClient<Type>("name")`。
  - **调用**：使用 `client.call(srv)`。注意这是**同步阻塞**的，拿到结果前程序不会往下跑。

### 3. CMakeLists.txt 关键配置

```cmake
# 1. 必须包含 message_generation
find_package(catkin REQUIRED COMPONENTS roscpp message_generation)

# 2. 声明服务文件
add_service_files(FILES AddTwoInts.srv)

# 3. 强制编译顺序（最重要：防止找不到生成的头文件）
add_dependencies(your_node ${${PROJECT_NAME}_EXPORTED_TARGETS})
```

***

## 一句话总结

**Service 是 ROS 中的同步 RPC 机制：通过** **`.srv`** **定义契约，Server 提供被动回调，Client 发起阻塞调用，它是 Topic 的重要补充，共同构成了 ROS 通信的基石。**
