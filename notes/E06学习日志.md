## 学习目标

理解 ROS 中的**动作通信（Action）**机制。Topic 适合流式数据，Service 适合一问一答，但当任务需要**很长时间**（如让机械臂运动到位），并且你还想**中途看进度、随时取消**时，Topic 和 Service 都不够用——这就是 Action 的用武之地。

***

## 关键问题及答案

### 1. 为什么机械臂运动到目标位姿这个场景，Topic 和 Service 都不适合？

**答**：
*   **Topic（广播）**：只管发指令，发完就不管了。你没法知道机械臂动到哪了，也没法拿到"运动完成"的反馈。就像朝对岸扔漂流瓶，不知道对方收到了没有。
*   **Service（一问一答）**：虽然是双向的，但 Client 调用 `call()` 后会**卡死等待**。机械臂运动需要好几秒，这几秒里 Client 什么都不能做，而且中途没法取消，万一要撞墙了也停不下来。

### 2. `.action` 文件为什么要分三段？每段由谁发给谁？

**答**：
三段对应了 Action 的三个核心环节：

| 段 | 内容 | 方向 | 理解 |
| :--- | :--- | :--- | :--- |
| **Goal**（目标） | "我要干什么" | Client → Server | 相当于 Service 的 Request |
| **Result**（结果） | "最后怎么样了" | Server → Client | 相当于 Service 的 Response |
| **Feedback**（反馈） | "现在到哪了" | Server → Client | **Action 独有**，Service 没有的能力 |

### 3. Preemption（取消）的本质是什么？Server 不检查会怎样？

**答**：
*   **本质**：Server 并不是被动的"收到信号就断电"，而是**主动检查**。Server 必须在自己的任务循环里定期调用 `isPreemptRequested()` 来"过问"Client 有没有想取消。
*   **不检查的后果**：Client 虽然发了取消，但 Server 根本不理，继续执行直到完成。取消操作变成了摆设，整个机制失效。

### 4. Action 底层的 5 个 Topic 分别承载什么？

**答**：
Action 是一个"包装好的高级接口"，底层实际由 5 个 Topic 组成：

| Topic 名称 | 作用 |
| :--- | :--- |
| `action_name/goal` | 传递 Goal（目标） |
| `action_name/cancel` | 传递取消请求 |
| `action_name/status` | 传递 Server 当前状态 |
| `action_name/feedback` | 传递进度反馈 |
| `action_name/result` | 传递最终结果 |

这就是为什么说 **Action 是 Topic + Service 的组合拳**。

### 5. 为什么取消时要调 `setPreempted()` 而不是 `setSucceeded()`？

**答**：
这两个状态对 Client 的 `getState()` 返回值完全不同：
*   **`setSucceeded()`** → Client 拿到状态 `SUCCEEDED`（任务正常完成）。
*   **`setPreempted()`** → Client 拿到状态 `PREEMPTED`（任务被中途取消）。
Client 可以通过判断状态来区分"正常结束"和"被取消"，做不同的后续处理。

## 核心知识点总结

### 三种通信机制对比

| 维度 | Topic | Service | Action |
| :--- | :--- | :--- | :--- |
| **方向** | 单向 | 双向 | 双向 + 持续 |
| **时机** | 实时流式 | 即时一问一答 | 长时间任务 |
| **中途反馈** | ❌ 无 | ❌ 无 | ✅ 有 Feedback |
| **可取消** | ❌ 否 | ❌ 否 | ✅ 是（Preemption）|
| **典型场景** | 传感器数据、控制指令 | 查状态、开关、计算 | 导航、机械臂运动、倒计时 |

### Action Server 核心方法

| 操作 | 方法 |
| :--- | :--- |
| 接收目标 | `registerGoalCallback(callback)` |
| 发反馈 | `publishFeedback(feedback)` |
| 任务成功 | `setSucceeded(result)` |
| 响应取消 | `isPreemptRequested()` + `setPreempted()` |
| 任务失败 | `setAborted(result)` |

### Action Client 核心方法

| 操作 | 方法 |
| :--- | :--- |
| 等 Server 上线 | `waitForServer()` |
| 发目标 | `sendGoal(goal, doneCb, activeCb, fbCb)` |
| 查状态 | `getState()` |
| 取消任务 | `cancelGoal()` |
| 阻塞等结果 | `waitForResult(timeout)` |

***

## 一句话总结

**Action 是对 Service 的增强升级——当任务耗时较长、需要进度反馈、支持中途取消时，Action 是唯一选择；它的三板斧（Goal → Feedback → Result）让 Client 不再当"瞎子"。**
