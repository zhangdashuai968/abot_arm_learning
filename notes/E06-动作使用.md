## 这节解决什么问题

Topic 是异步单向的。Service 是同步一问一答的。

但如果有一个任务：**"让机械臂从当前位置运动到目标关节角"**，这个操作需要好几秒，期间你希望：
1. 知道当前进度（feedback）
2. 随时可以取消（preemption）
3. 任务完成后收到最终结果（result）

Topic 和 Service 都做不到。你需要 **Action**。

---

## Action 的概念模型（O'Kane 的核心描述）

```
Client                                Server
  |                                     |
  |-------- Goal (关节角 = ...) ------->|
  |                                     | (开始执行)
  |<--- Feedback (进度 20%) -----------|
  |<--- Feedback (进度 50%) -----------|
  |<--- Feedback (进度 80%) -----------|
  |                                     |
  |-------- Cancel  ------------------>|  ← Preemption
  |                                     |
  |<--- Result (成功/失败) ------------|
```

**Action 是 Topic + Service 的超级组合**：
- 用 topic 传递 goal、feedback、result
- 用 service 实现 cancel
- 底层基于 **actionlib** 库

---

## 三种通信机制对比

| 维度 | Topic | Service | Action |
|---|---|---|---|
| 方向 | 单向 | 双向 | 双向 + 持续 |
| 时机 | 实时 | 即时 | 长时间 |
| 反馈 | 无 | 无 | 有（feedback） |
| 可取消 | 否 | 否 | 是（preemption） |
| 底层实现 | TCP/UDP | TCP | actionlib（Topic + Service） |

> O'Kane 金句：**"Use actions for anything that might take a while and might need to be interrupted."**

## Action 定义文件（.action）

创建 `action/Timer.action`：

```
# Goal
int32 duration   # 倒计时总秒数
---
# Result
int32 elapsed    # 实际经过的秒数
---
# Feedback
int32 remaining  # 剩余秒数
```

> `.action` 文件分为三段，用 `---` 分隔。顺序固定：**Goal → Result → Feedback**。

## 编写 Action Server（C++）

O'Kane 第 6 章使用 `actionlib::SimpleActionServer`，这是最简洁的写法：

```cpp
#include <ros/ros.h>
#include <actionlib/server/simple_action_server.h>
#include <custom_actions/TimerAction.h>

typedef actionlib::SimpleActionServer<custom_actions::TimerAction> Server;

void execute(const custom_actions::TimerGoalConstPtr &goal, Server *as) {
    ros::Rate rate(1);
    custom_actions::TimerFeedback feedback;

    // 发送 feedback：倒计时
    for (int i = goal->duration; i > 0; i--) {
        // 检查 client 是否发了 cancel 请求
        if (as->isPreemptRequested()) {
            ROS_WARN("Timer preempted!");
            as->setPreempted();  // 通知 client 已取消
            return;
        }

        feedback.remaining = i;
        as->publishFeedback(feedback);

        rate.sleep();
    }

    // 任务完成：设置 result
    custom_actions::TimerResult result;
    result.elapsed = goal->duration;

    as->setSucceeded(result);  // 通知 client 成功完成
}

int main(int argc, char **argv) {
    ros::init(argc, argv, "timer_server");

    ros::NodeHandle nh;
    Server server(nh, "timer", false);  // false = 不自动启动
    server.registerGoalCallback(
        boost::bind(&execute, _1, &server));

    server.start();  // 手动启动
    ros::spin();
}
```

**Server 核心要点**：

| 操作 | 方法 | 含义 |
|---|---|---|
| 接收 goal | `registerGoalCallback(callback)` | 新 goal 到来时触发 |
| 发 feedback | `publishFeedback(feedback)` | 通知 client 当前进度 |
| 完成 | `setSucceeded(result)` | 任务成功 |
| 取消 | `isPreemptRequested()` + `setPreempted()` | 响应取消请求 |
| 失败 | `setAborted(result)` | 任务异常终止 |

> O'Kane 详细解释了 `boost::bind`：因为 `registerGoalCallback` 期望一个无参回调，但我们需要传 `server` 对象进去，所以用 `bind` 绑定参数。

## 编写 Action Client（C++）

```cpp
#include <ros/ros.h>
#include <actionlib/client/simple_action_client.h>
#include <custom_actions/TimerAction.h>

typedef actionlib::SimpleActionClient<custom_actions::TimerAction> Client;

void feedbackCb(const custom_actions::TimerFeedbackConstPtr &fb) {
    ROS_INFO_STREAM("Remaining: " << fb->remaining);
}

int main(int argc, char **argv) {
    ros::init(argc, argv, "timer_client");

    Client client("timer", true);  // true = 自动启动

    ROS_INFO("Waiting for timer server...");
    client.waitForServer();  // 阻塞直到 server 上线

    custom_actions::TimerGoal goal;
    goal.duration = 5;

    // 发送 goal + 注册 feedback 回调
    client.sendGoal(goal,
        Client::SimpleDoneCallback(),  // 完成后的回调
        Client::SimpleActiveCallback(), // 激活后的回调
        &feedbackCb);                   // feedback 回调

    ROS_INFO("Goal sent!");

    // 等 3 秒后取消
    ros::Duration(3.0).sleep();

    if (client.getState() == actionlib::SimpleClientGoalState::ACTIVE) {
        ROS_INFO("Canceling goal...");
        client.cancelGoal();
    }

    ros::spin();
}
```

**Client 核心要点**：

| 操作 | 方法 | 含义 |
|---|---|---|
| 连接 | `waitForServer()` | 等待 action server 上线 |
| 发 goal | `sendGoal(goal, doneCb, activeCb, fbCb)` | 发送目标 |
| 查状态 | `getState()` | PENDING / ACTIVE / SUCCEEDED 等 |
| 取消 | `cancelGoal()` | 请求取消 |
| 阻塞等结果 | `waitForResult(timeout)` | 同步等待版本 |

## Action 的 5 个内部 Topic

O'Kane 指出，actionlib 底层自动创建了 5 个 topic：

```
action_name/goal        ← action 的 goal topic
action_name/cancel      ← action 的 cancel topic  
action_name/status      ← action server 状态
action_name/feedback    ← 进度反馈
action_name/result      ← 最终结果
```

你不需要手动操作这些 topic，`SimpleActionServer`/`SimpleActionClient` 封装好了。

## Preemption 的正确处理

O'Kane 在 Chapter 6 中反复强调一个坑：

```cpp
void execute(const GoalConstPtr &goal, Server *as) {
    for (int i = 0; i < 100000; i++) {
        // ❌ 坏写法：做大量工作但不检查 preemption
        heavy_work();

        // ✅ 好写法：定期检查
        if (as->isPreemptRequested()) {
            cleanup_before_exit();  // 清理工作
            as->setPreempted();
            return;
        }
    }
}
```

**原则**：你的 execute 回调里，每隔一段工作就要检查一次 `isPreemptRequested()`，否则 client 发了 cancel 你也不会停。

## 常见错误

| 错误 | 原因 |
|---|---|
| `waitForServer()` 一直不返回 | server 还没启动，或 action name 不匹配 |
| feedback 收不到 | client 侧忘记注册 feedback 回调 |
| cancel 不生效 | server 侧每次循环都忘记调 `isPreemptRequested()` |
| 链接错误 `undefined reference to actionlib::...` | CMakeLists 里没 `target_link_libraries(xxx ${catkin_LIBRARIES})` |

---

## 一句话总结

**Action 是 ROS 中处理长时间任务的机制：Goal → Feedback → Result，支持 Preemption；底层是 actionlib 库，Server 定期检查是否被取消是关键。**
