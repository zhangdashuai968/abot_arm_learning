## 这节解决什么问题

你的机器人上有多个串口设备——IMU、激光雷达、底盘控制器、机械臂驱动。它们都插在 USB 口上，Linux 按**插入顺序**分配设备名：

```
第一次开机：  IMU → /dev/ttyUSB0   Lidar → /dev/ttyUSB1
第二次开机：  Lidar → /dev/ttyUSB0   IMU → /dev/ttyUSB1  ← 全乱了！
```

你在 launch 文件里写死了 `/dev/ttyUSB0` 是 IMU，结果第二次开机它变成了激光雷达，程序崩溃。**你需要 udev 规则把设备名固定住。**

---

## 核心原理

udev 是 Linux 的设备管理器。它可以根据设备的**唯一属性**（序列号、厂商 ID、产品 ID）给设备分配一个**固定的符号链接名**。

```
物理属性（不变）      符号链接（你自己定的，永远不变）
USB 序列号 xxx  →     /dev/abot_imu
USB 序列号 yyy  →     /dev/abot_lidar
```

ROS 节点配置中永远使用符号链接名，不管实际是 ttyUSB0 还是 ttyUSB7。

---

## 1. 找到设备的唯一属性

```bash
# 插上设备后，列出所有 ttyUSB 设备
ls -l /dev/ttyUSB*

# 找到目标设备的 kernel 名称（比如 ttyUSB0），查它的属性
udevadm info --name=/dev/ttyUSB0 --attribute-walk

# 在输出中搜索 ID_SERIAL_SHORT、idVendor、idProduct 等字段
# 例如某个 IMU：
#   ATTRS{idVendor}=="0403"
#   ATTRS{idProduct}=="6001"
#   ATTRS{serial}=="A904M0IF"
```

> `--attribute-walk` 会列出从设备节点一路向上到 USB 控制器的所有属性。通常用最底层的 `ATTRS{idVendor}` + `ATTRS{idProduct}` 组合，或者用 `ATTRS{serial}`（序列号）最精确。

---

## 2. 编写 udev 规则

创建 `/etc/udev/rules.d/99-abot-serial.rules`：

```bash
sudo nano /etc/udev/rules.d/99-abot-serial.rules
```

写入规则：

```
# abot RPLidar A3M12
SUBSYSTEM=="tty", ATTRS{idVendor}=="10c4", ATTRS{idProduct}=="ea60", SYMLINK+="abot_lidar"

# abot IMU N100
SUBSYSTEM=="tty", ATTRS{idVendor}=="0403", ATTRS{idProduct}=="6001", ATTRS{serial}=="A904M0IF", SYMLINK+="abot_imu"

# abot 底盘控制器
SUBSYSTEM=="tty", ATTRS{idVendor}=="1a86", ATTRS{idProduct}=="7523", SYMLINK+="abot_base"
```

**每条规则的含义**：

| 字段 | 含义 |
|---|---|
| `SUBSYSTEM=="tty"` | 匹配 tty 类设备 |
| `ATTRS{idVendor}=="0403"` | USB 厂商 ID（可用 `lsusb` 查看） |
| `ATTRS{idProduct}=="6001"` | USB 产品 ID |
| `ATTRS{serial}=="A904M0IF"` | 设备序列号（同时有多个同型号设备时必用） |
| `SYMLINK+="abot_imu"` | 创建符号链接 `/dev/abot_imu` |

> `==` 是比较（匹配），`+=` 是添加（创建），`=` 是赋值（覆盖）。**不要用单个 `=` 做比较！**

---

## 3. 使规则生效

```bash
sudo udevadm control --reload-rules
sudo udevadm trigger

# 重新插拔设备，验证符号链接是否创建成功
ls -l /dev/abot_*
# 输出应类似：
# lrwxrwxrwx 1 root root 7 Mar 30 10:00 /dev/abot_imu -> ttyUSB0
# lrwxrwxrwx 1 root root 7 Mar 30 10:00 /dev/abot_lidar -> ttyUSB1
```

---

## 4. 权限问题

默认创建的 `/dev/abot_imu` 权限只允许 root 用户访问。ROS 节点不能用 root 跑，需要改权限：

```
# 方法 1：在 udev 规则中加 MODE（推荐）
SUBSYSTEM=="tty", ATTRS{idVendor}=="0403", ATTRS{idProduct}=="6001", ATTRS{serial}=="A904M0IF", SYMLINK+="abot_imu", MODE="0666"

# 方法 2：把用户加入 dialout 组
sudo usermod -a -G dialout $USER
# 注销再登录才生效
```

> O'Kane 在涉及传感器驱动的章节多次提到这个坑：**文件权限导致 `open()` 失败是最常见的"传感器莫名其妙不工作"的原因。**

---

## 5. 在 ROS 中使用

在 launch 文件或节点代码中使用符号链接名：

```xml
<!-- launch 文件中的激光雷达配置 -->
<node name="rplidar_node" pkg="rplidar_ros" type="rplidarNode">
    <param name="serial_port" value="/dev/abot_lidar" />
    <param name="serial_baudrate" value="256000" />
</node>
```

```cpp
// C++ 节点中打开串口
#include <fcntl.h>
#include <termios.h>

int fd = open("/dev/abot_imu", O_RDWR | O_NOCTTY);
if (fd < 0) {
    ROS_FATAL("Cannot open /dev/abot_imu. Check udev rules.");
    return 1;
}
```

## 完整 abot 常用设备 idVendor/idProduct 速查

| 设备 | 接口 | idVendor | idProduct | 符号链接 | 波特率 |
|---|---|---|---|---|---|
| RPLIDAR A3M12 | USB | 10c4 | ea60 | `/dev/abot_lidar` | 256000 |
| IMU N100 | USB | 0403 | 6001 | `/dev/abot_imu` | 115200 |
| Zeus S2 底盘 | USB | 1a86 | 7523 | `/dev/abot_base` | 115200 |
| ROCR6 机械臂 | Ethernet | — | — | — | — |

> **注意**：ROC R6 走的是以太网（TCP/IP），不经过 udev，用的是固定 IP。以上 idVendor/idProduct 为典型值，实际需要在自己机器上 `udevadm info` 确认。

## 调试技巧

```bash
# 实时监控 udev 事件（插拔设备时能看到匹配了哪条规则）
sudo udevadm monitor --environment

# 测试某条规则是否匹配
sudo udevadm test $(udevadm info --query=path --name=/dev/ttyUSB0)

# 查看设备是否被占用
sudo lsof /dev/ttyUSB0
```

## 常见错误

| 错误 | 原因 |
|---|---|
| 符号链接没出现 | 规则中的 `ATTRS{...}` 值写错了，用 `udevadm info --attribute-walk` 重新确认；或 `==` 写成 `=` |
| `Permission denied` | 没加 `MODE="0666"` 或用户不在 `dialout` 组 |
| 规则修改后不生效 | 记得执行 `sudo udevadm control --reload-rules` 和 `sudo udevadm trigger` |
| 两个同型号设备无法区分 | 用 `ATTRS{serial}`（序列号）代替 idVendor/idProduct |

---

## 一句话总结

**udev 规则 = 用唯一属性锁定设备 + 创建固定符号链接 + 设好权限；所有 ROS 配置中都用符号链接名，"开机后 ttyUSB 编号不可靠"这个坑永远消失。**
